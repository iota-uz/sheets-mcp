# IOTA MCP — Автоматизация финансовой таблицы

## Что это

Проект автоматизирует ведение Google Sheets таблицы **Finances_2024_2025** компании IOTA.
Claude читает банковские выписки (.xlsx) и скриншоты переводов, извлекает данные и записывает их в нужные листы таблицы через собственный MCP сервер.

## MCP-инструменты

| Tool | Что делает |
|------|------------|
| `sheets_describe` | Метаданные листа: sheetId, headerRow, rowCount, headers (массив `{index, letter, text}`). Без аргумента — список всех листов. |
| `sheets_exec` | JS в sandbox'е с `sheets` API. `dryRun: true` → captured planned batchUpdate без отправки. |
| `discord_read_messages` | Читает сообщения Discord #finances. |

В `sheets_exec` доступно:
```js
const расходы = await sheets.sheet("расходы");      // headerRow по умолчанию = 1
const ддс     = await sheets.sheet("ддс", { headerRow: 3 });

// Records keyed by header text exactly. Formulas начинаются с "=" и могут
// использовать placeholders {row} и {col:HeaderName} (резолвятся в runtime).
await расходы.insert({
  "Дата": "06.04.2026",
  "Категория": "Сервера и подписки",
  "Сумма": 292.50,
  "Валюта": "USD",
  "Курс": '=SWITCH({col:Валюта}{row}; "USD";1; ...)',
  "Сумма ($)": "={col:Сумма}{row}/{col:Курс}{row}",
  "Расчетный период": "01.04.2026",
  "Исходящий счет": "Tenge Mastercard",
  "Описание": "Blacksmith Software",
}, { idempotencyKey: "06.04.2026|292.5|Blacksmith" });

await расходы.find({ "Категория": "ФОТ" });
await расходы.update({ where: { "Описание": "X" }, set: { "Исходящий счет": "Y" }});
await расходы.delete({ where: { "Описание": "X" }});
await расходы.format({ where: { "Категория": "ФОТ" }, set: { backgroundColor: "#fff3cd" }});

// Set up dropdown / strict validation (rules live IN the sheet — source of truth)
await расходы.setValidation("Категория", {
  type: "ONE_OF_LIST",
  values: ["ФОТ", "Налоги", "EAI Google", ...],
  strict: true,
});
```

**Важно:** MCP не знает о конкретных колонках или категориях этой таблицы. Все имена колонок (`Дата`, `Категория`, `Курс`, ...) и enum-значения (`ФОТ`, `USD`, ...) — runtime данные от агента. Точные написания смотри в этом CLAUDE.md и в SKILL.md. Заголовки можно проверить через `sheets_describe sheet="расходы"`.

# Листы таблицы и что в них записывать

В таблице Finances_2024_2025 есть три листа, в которые мы пишем:

## 1. Лист "расходы"

**Что сюда идёт:** всё, что компания потратила — зарплаты, налоги, аренда, реклама, комиссии банка и т.д.

| Колонка | Что записывать |
|---------|---------------|
| B — Дата | Дата платежа (ДД.ММ.ГГГГ) |
| C — Категория | ФОТ / Налоги / Маркетинг и реклама / EAI Google / Аренда офис / Комиссия банка / Офисные припасы / Наемные рабоники вне штата / Сервера и подписки / Другое |
| D — Сумма | Сумма в исходной валюте |
| E — Курс | *Считается автоматически* (Google Finance) |
| F — Валюта | `USD` или `SOM` |
| G — Сумма ($) | *Считается автоматически* (D / E) |
| H — Расчётный период | *Считается автоматически* |
| I — Исходящий счет | Откуда ушли деньги. Для выписок с р/с всегда `Kapital UZS (р/с)` |
| J — Сотрудник | ФИО (если это зарплата/премия) |
| K — Описание | Кратко суть: "ЗП март", "Реклама Autostrada.uz", "НДФЛ" |

**Как определяется категория (для выписок):**

| Категория | Как распознать в "Назначении платежа" |
|-----------|--------------------------------------|
| **ФОТ** | Контрагент `BEFORMED`, слова "ЗП", "зарплата", "премиальные", "премия", "неиспользованный отпуск" |
| **Маркетинг и реклама** | "реклама", "размещение", "публикация", "пакет сайт", "партнерство ассоциация" |
| **Аренда офис** | "аренда помещения", "аренда офис" |
| **Офисные припасы** | "A3 LIST", "ROLLAB", "стенд", "полиграф" |
| **Наемные рабоники вне штата** | "бухгалтерские" |
| **Налоги** | "обслуживание пассивных счетов", "пенсия", "НДФЛ", "импорт услуг", узб. налоговые термины |
| **Комиссия банка** | "тариф", "SWIFT", "комиссия инобанка", "за документ S=" |
| **Сервера и подписки** | SaaS-подписки и хостинг — см. раздел ниже про карты |
| **Другое** | Всё, что не подошло под категории выше |

---

## 2. Лист "платежи"

**Что сюда идёт:** деньги, которые поступили в компанию (оплата от клиентов за проекты).

| Колонка | Что записывать |
|---------|---------------|
| A — Итерация | `Проект-N` (например `Granite-15`). Номер берётся из листа "итерации" |
| B — Проект | `Granite` / `Takhir` / `EAI_Website` / `Reklama EAI` / `EAI_App` / `SHY ELD` / `Zender` |
| C — Сумма | Сумма платежа |
| D — Категория | `Разработка ПО под ключ` / `Поддержка и мелкие доработки` / `Консалтинг и продажа часов` / `Другие источники дохода` |
| E — Курс | *Считается автоматически* (Google Finance) |
| F — Валюта | `USD` или `SOM` |
| G — Сумма ($) | *Считается автоматически* |
| H — Дата | Дата поступления (ДД.ММ.ГГГГ) |
| I — Расчётный период | *Считается автоматически* |
| J — Входящий счёт | Куда поступили деньги: `Kapital UZS (р/с)`, `Kapital USD (р/с)`, `Наличные USD` и др. |
| K — Комментарий | `На р/с`, `оплата 30%`, `оплата 100%` и т.д. |

**Как определяется проект (для выписок с р/с):**

В банковской выписке платежи — это строки с кредитом > 0 (деньги пришли на счёт). Проект определяется по описанию:

| Проект | Как распознать |
|--------|---------------|
| **Granite** | В описании есть "АИС GRANITE" или "разработка и внедрение автоматизированной системы" |
| **Takhir** | "консалтинговые услуги по цифровизации" + сумма 101 000 000 |
| **Reklama EAI** | "цифровое курирование" или "консалтинг" + сумма > 300 000 000 |
| **EAI_Website** | "техническая поддержка, сопровождение веб" или "консалтинг" + средняя сумма |

**Важно: перед добавлением платежа нужно создать итерацию!**

### Лист "итерации" (вспомогательный)

Каждый платёж привязан к итерации проекта. Перед записью в "платежи" нужно:

1. Узнать последний номер итерации для проекта (например Granite-14)
2. Создать новую строку в "итерации": Проект = `Granite`, ID = `Granite-15`, # = `15`, Оплачено = сумма, Остаток = сумма, Валюта = `SOM`
3. Только потом записать платёж в "платежи" с этой итерацией

Скрипт `import-statement.mjs` делает это автоматически.

## 3. Лист "ддс"

**Что сюда идёт:** финансовая помощь — переводы с расчётного счёта на личную карту начальника (Хайдаров Диёр). Это НЕ расход и НЕ платёж.

| Колонка | Что записывать |
|---------|---------------|
| A — Дата | Дата операции (ДД.ММ.ГГГГ) |
| B — Сумма | Сумма в UZS |
| C — Исходящий счет | Всегда `UzCard (Kapital)` |
| D — Входящий счет | Всегда `Uzum Visa UZS` |
| E — Тип | Всегда `P2P` |
| F — Расчётный период | *Считается автоматически* |
| G — Обменный курс | Обычно пусто |
| H — Комментарий | Обычно пусто |

**Как распознать ддс в выписке:** описание содержит "фин займ" или "фин помощь", или контрагент — "Транзитный счёт ... Самарканд".

# Как обрабатывать данные

## Вариант 1: Excel-выписка с расчётного счёта

Пользователь кидает `.xlsx` файл в папку `reports/` и говорит **"обработай отчёт"**.

Обработка этого кейса делается по skill'у **`import-bank-statement`** (`.claude/skills/import-bank-statement/SKILL.md`). Skill автоматически подхватится и я пошагово пройду workflow:

1. Парсим xlsx через `mcp/excel.mjs` → получаем структурированные строки
2. Классифицируем каждую по смыслу (LLM, не regex): кредит → платёж, "фин займ" → ддс, дебет → расход
3. Для платежей — создаём новую итерацию (находим max номер по проекту, +1)
4. Показываем dry-run → ждём подтверждения пользователя
5. Пишем одним JS-скриптом через `sheets_exec` — `Table.insert` идемпотентен по primary key, формулы курса/USD автозаполняются из схемы.

Детали правил классификации и порядка записи — в `SKILL.md` скилла.

## Вариант 2: Скриншот перевода

Пользователь кидает скриншот в чат или в папку `reports/` и говорит **"обработай расходы"** (или "обработай платежи") с комментарием.

1. Прочитать изображение
2. Извлечь визуально: дату, сумму, валюту, контрагента, описание
3. Записать данные в соответствующий лист (расходы / платежи / ддс) через MCP

## Вариант 3: Discord-канал #finances

В Discord-сервере IOTA есть канал `#finances`, куда сотрудники кидают скриншоты переводов с кратким описанием (например: "Комиссия банка" + фото чека). Это **всегда расходы**.

Пользователь говорит **"обработай дискорд"**.

1. Вызвать `discord_read_messages` → получить список сообщений с текстом и путями к скачанным скринам
2. Для каждого сообщения: прочитать скрин → извлечь дату, сумму, валюту
3. Записать каждый расход в таблицу через MCP
4. Удалить скачанные скрины через `rm` (не захламлять папку)

**Важно: названия счетов должны точно совпадать с листом "Счета".** Посмотри там список счетов и пиши один в один (например `Uzum Visa UZS`, а НЕ `Uzum UZS`). Несовпадения всплывут как "орфаны" в сверке баланса.

## Вариант 4: Сверка баланса

В конце месяца пользователь говорит **"сверь баланс"** / **"посчитай расхождение"** / **"проверь счета"** / **"откуда разница"**.

Сверкой занимается **skill `reconcile-finances`** (`.claude/skills/reconcile-finances/`). Он подхватится автоматически по фразам-триггерам — читай `SKILL.md` в папке skill'а для деталей по workflow.

Вкратце:
- Skill запускает `node ${CLAUDE_SKILL_DIR}/scripts/reconcile.mjs --json`
- Скрипт читает "Счета", считает поток по каждому счёту, ищет орфанов и подозрительные паттерны
- Результат интерпретируется по приоритету: орфаны → паттерны → кросс-проверка с выпиской из `reports/` → топ-5 крупнейших

Ручной запуск скрипта (если нужно):
```bash
node .claude/skills/reconcile-finances/scripts/reconcile.mjs                      # текущий месяц, текст
node .claude/skills/reconcile-finances/scripts/reconcile.mjs --json               # для программного разбора
node .claude/skills/reconcile-finances/scripts/reconcile.mjs --period=01.03.2026  # другой месяц
```

# Справка

## Счёт (для расходов из выписки)
Если данные взяты из банковской выписки с р/с → счёт всегда **`Kapital UZS (р/с)`**.

## Список счетов
Названия счетов живут в листе **"Счета"** (колонка A). При записи транзакций используй **точно** это написание (регистр, пробелы, скобки). Частые примеры: `Kapital UZS (р/с)`, `Kapital USD (р/с)`, `Uzum Visa UZS`, `UzCard (Kapital)`, `Tenge Humo`, `Наличные USD`, `Наличные UZS`.

## Запуск MCP сервера вручную (если не подключился)
```bash
node mcp/server.mjs
```

## Настройка валидации (one-off через `sheets_exec`)

Чтобы лист "знал" список разрешённых значений для enum-колонки (Категория, Валюта, Тип, и т.д.) — выставь dropdown через `sheet.setValidation`. Это ставит data validation НА ЛИСТЕ — теперь и UI, и MCP проверяют ввод против этого списка. Источник истины — сама таблица, не код.

Пример (запустить один раз через `sheets_exec`):
```js
const расходы = await sheets.sheet("расходы");
await расходы.setValidation("Категория", {
  type: "ONE_OF_LIST",
  values: ["ФОТ", "Налоги", "Маркетинг и реклама", "EAI Google",
           "Аренда офис", "Комиссия банка", "Офисные припасы",
           "Наемные рабоники вне штата", "Сервера и подписки", "Другое"],
  strict: true,
});
await расходы.setValidation("Валюта", {
  type: "ONE_OF_LIST",
  values: ["USD", "SOM", "EUR", "RUB"],
  strict: true,
});
```
После этого попытка записать значение не из списка вернёт ошибку валидации до отправки в API.
