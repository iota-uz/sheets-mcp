# Сверка баланса IOTA

Skill для поиска и устранения расхождений между фактическим балансом и тем что насчитала таблица.

## Когда применять

Запускай, если пользователь:
- Говорит "сверь баланс", "посчитай расхождение", "проверь счета", "откуда разница"
- Просит проверить финансы в конце месяца
- Упоминает расхождение в листе "Счета"

**НЕ запускай**, если пользователь просто обсуждает отдельный платёж или расход — это не сверка.

## Шаг 1 — запуск скрипта

```bash
node ${CLAUDE_SKILL_DIR}/scripts/reconcile.mjs --json
```

С указанием периода:
```bash
node ${CLAUDE_SKILL_DIR}/scripts/reconcile.mjs --json --period=01.03.2026
```

Если пользователь сказал "за март" — преобразуй в `--period=01.03.2026`. Без уточнений — текущий месяц.

Скрипт выведет JSON с полями: `period`, `summary` (разница), `perAccount`, `orphans`, `suspects`, `topLargestInPeriod`, `inPeriodCounts`.

## Шаг 2 — оценка разницы

Смотришь `summary.differenceUsd`.

**Разница = Фактический − Таблица.**

| Диапазон | Что значит | Что делать |
|----------|------------|------------|
| `−$2000 < Δ < +$7000` | В пределах нормы (это "около нуля" для этой таблицы) | Сообщить "всё сходится, дальше копать не надо". Показать только разницу и остановиться. |
| `Δ > +$7000` (большой плюс) | Фактический > Таблицы. Реальности больше, в таблице **не хватает доходов** → пропущен **платёж** | Идти в Шаг 3, потом в Шаг 5 с прицелом "ищи кредит > 0 которого нет в листе 'платежи'" |
| `Δ < −$2000` (большой минус) | Фактический < Таблицы. Реальности меньше, в таблице **не хватает расходов** → пропущен **расход или ддс** | Идти в Шаг 3, потом в Шаг 5 с прицелом "ищи дебет > 0 которого нет в листах 'расходы'/'ддс'" |

## Шаг 3 — орфаны (самый частый виновник)

Если `orphans[]` непустой:

1. Покажи пользователю список орфанов с их потоком денег (`incomingUsd`, `outgoingUsd`, `transfersInUsd`, `transfersOutUsd`, количество транзакций).
2. Для каждого орфана найди ближайший настоящий счёт из `perAccount` (скорее всего пользователь имел в виду его — типичный кейс "Uzum UZS" вместо "Uzum Visa UZS").
3. Спроси: "Исправить название `X` → `Y`? Это затронет N строк."
4. **Жди подтверждения пользователя.** Только после "да" — найди через `sheets_read` строки с этим орфан-счётом и исправь через `sheets_update` ячейки в колонке I (для расходов) / J (для платежей) / C или D (для ддс).

После исправления — предложи запустить сверку ещё раз.

## Шаг 4 — подозрительные паттерны

Если `suspects[]` непустой — покажи топ-3 с человеческой интерпретацией:

| `reason` | Как объяснять пользователю |
|----------|---------------------------|
| `exact_match` | "Сумма точно равна разнице — возможно запись лишняя или пропущена" |
| `doubled` | "Сумма × 2 = разница — возможно дубликат" |
| `halved` | "Сумма / 2 = разница — возможно записано в 2 раза больше" |
| `currency_error` | "SOM-сумма близка к разнице в $ — возможно перепутана валюта" |
| `decimal_shift` | "Отличается в 10 раз — возможно лишний/пропущенный ноль" |

Для каждого suspect показывай: лист + номер строки + причина + описание (первые 100 символов). Предложи посмотреть через `sheets_read` конкретную строку.

## Шаг 5 — кросс-проверка с выпиской (если Шаги 3–4 не помогли)

Если разница всё ещё большая и направление определено (см. Шаг 2):

1. Найди xlsx-файлы в `reports/` через `Glob`:
   ```
   reports/*.xlsx
   ```
2. Если файлов несколько — спроси пользователя какую выписку брать.
3. Прочитай выписку через `Read` (отображается как таблица).
4. Для направления:
   - **Плюс** (ищем пропущенный платёж) — отфильтруй строки где "Обороты по кредиту" > 0
   - **Минус** (ищем пропущенный расход) — отфильтруй строки где "Обороты по дебету" > 0
5. Сравни эти строки со списком транзакций в нужном листе (прочитай через `sheets_read` за нужный период).
6. Покажи пользователю: "Вот строка из выписки на X сум от DD.MM.YYYY, которой не хватает в таблице". Предложи добавить — но **сам не добавляй без подтверждения**.

## Шаг 6 — топ-5 крупнейших (последняя надежда)

Если ничего не нашлось автоматически — покажи `topLargestInPeriod` и скажи: "Автоматика не нашла виновника. Проверь эти 5 крупнейших транзакций вручную, сравни с выпиской — возможно там опечатка в сумме."

## Чего НЕ делать

- ❌ Не менять ячейки в листе **"Счета"** — они правятся вручную пользователем по факту остатков в банках.
- ❌ Не исправлять орфанов пачкой без подтверждения **каждого** — могут быть легитимные редкие счета.
- ❌ Не удалять подозрительные строки автоматически.
- ❌ Не добавлять пропущенные транзакции из выписки без явного "да".
- ❌ Не копать, если `|Δ| < $2000` — это норма, копеечный шум таблицы.
